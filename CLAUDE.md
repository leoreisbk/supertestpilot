# TestPilot — Claude Code Instructions

## What This Project Is

TestPilot is a Kotlin Multiplatform (KMM) SDK for AI-driven end-to-end mobile testing. Tests are written in plain English; GPT-4 interprets the accessibility tree and executes steps autonomously. Supports iOS (XCTest) and Android (UIAutomator).

## Tech Stack

- **Core SDK**: Kotlin Multiplatform 1.8, Gradle (JDK 11+)
- **AI client**: `com.aallam.openai:openai-client:3.1.1` (aallam)
- **HTTP**: Ktor 2.2.4
- **iOS wrapper**: Swift + SPM, consumes KMM-built XCFramework via cinterop
- **Android**: UIAutomator 2.2.0, min SDK 29

## Project Structure

```
sdk/testpilot/src/
  commonMain/   ← Platform-agnostic logic: Runner, prompts, Config, TestActor
  androidMain/  ← UIAutomator-backed snapshot + actor
  iosMain/      ← XCTest cinterop + .def files, iOS snapshot + actor
swift-wrapper/  ← Thin Swift API on top of KMM XCFramework
scripts/        ← build_ios_sdk.sh
Package.swift   ← SPM manifest (points at KMM-built XCFramework)
```

## Build Commands

```bash
# Build iOS XCFramework (run from repo root)
scripts/build_ios_sdk.sh

# Build Android AAR
cd sdk && ./gradlew testpilot:assembleDebug

# Build release XCFramework
scripts/build_ios_sdk.sh testpilot:assembleTestPilotSharedReleaseXCFramework
```

## Key Files

- `sdk/testpilot/src/commonMain/…/runtime/Runner.kt` — core AI loop
- `sdk/testpilot/src/commonMain/…/runtime/prompts/` — GPT-4 prompts
- `sdk/testpilot/src/commonMain/…/runtime/Config.kt` — ConfigBuilder API
- `swift-wrapper/XCTestCase+Automation.swift` — iOS public API
- `docs/DEVELOPMENT.md` — full build & install guide

## Code Conventions

- Kotlin: standard KMM patterns, `expect`/`actual` for platform abstractions
- Swift: extensions on `XCTestCase`, `@MainActor` async
- Commit style: `type(scope): description` (e.g. `feat(KMM): add persistence api`)
- No tests directory in this repo — TestPilot IS the test framework

## Important Notes

- The `.def` files in `iosMain/` are generated by `build_ios_sdk.sh` from `.templ` files — do not edit `.def` files directly
- `Package.swift` references the XCFramework at a build output path; it won't resolve until the iOS SDK is built
- Config requires at minimum an `apiKey`; all other fields have defaults (maxTokens=200, maxSteps=10, temperature=0.0)
