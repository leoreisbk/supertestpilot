Building for Development
===

TestPilot is a Kotlin Multiplatform (KMM) project that supports iOS, Android, and Web. Most of the framework logic is shared across platforms via the `commonMain` source set; platform-specific code lives in `iosMain`, `androidMain`, and a JVM-based web runner.

---

## Architecture Overview

```
sdk/testpilot/src/
  commonMain/   ← Platform-agnostic logic: Analyst, TestAnalyst, prompts, Config
  androidMain/  ← UIAutomator-backed snapshot + actor
  iosMain/      ← XCTest cinterop + iOS snapshot + actor + CachingAIClient

swift-wrapper/  ← Thin Swift API over the KMM XCFramework
scripts/        ← build_ios_sdk.sh (builds XCFramework, populates .def files)
mac-app/        ← macOS companion app (SwiftUI)
testpilot       ← CLI entry point (bash script)
```

The CLI (`./testpilot`) and the Mac app are the two end-user interfaces. Both rely on **pre-built artifacts** downloaded automatically on first launch:

- `ios` artifact — XCFramework + harness runner (used by iOS runs)
- `web` artifact — JRE + JAR (used by Web runs)

Android runs use the Gradle-built AAR directly via ADB.

---

## Requirements

### All platforms
- macOS (for building and running the Mac app or CLI)
- A valid AI API key (Anthropic, OpenAI, or Google Gemini)

### iOS
- Xcode 15+
- iOS Simulator or physical iPhone/iPad

### Android
- Android Studio + Android SDK (API 29+)
- ADB in `$PATH`
- Android Emulator or physical device with USB debugging enabled

### Web
- No extra install — the bundled JRE is downloaded automatically by the Mac app or CLI

---

## Mac App

The Mac app is the recommended way to run analyses and tests for non-technical users. On first launch it downloads the required artifacts (XCFramework + web runner) automatically and stores them under `~/.testpilot/`.

To build and run locally:

```bash
open mac-app/TestPilot.xcodeproj
```

Run the `TestPilotApp` scheme. In DEBUG builds the artifact download step is skipped, so the locally built XCFramework is used directly.

### Releasing

Releases are built via Fastlane (see `mac-app/fastlane/Fastfile`). A GitHub Actions workflow at `.github/workflows/release.yml` triggers on version tags and produces a signed, notarized DMG uploaded to the GitHub release.

---

## CLI

`./testpilot` is a self-contained bash script — no installation required. It uses the same pre-built artifacts as the Mac app (`~/.testpilot/`). On first run, if the artifacts are missing, it downloads them from the latest GitHub release.

```bash
# Analyze an iOS app
./testpilot analyze --platform ios --app "MyApp" --objective "describe the onboarding flow"

# Test a condition on a web URL
./testpilot test --platform web --url https://example.com --objective "login button is visible"
```

See `docs/guia-analise-ux.md` for full usage instructions.

## Building from Source

### iOS XCFramework

The XCFramework is normally downloaded automatically. To build it locally (required if you're modifying the SDK):

```bash
# From the repo root
scripts/build_ios_sdk.sh
```

This populates `.def` files from `.templ` templates (using the path from `xcode-select --print-path`) and runs the Gradle task that produces the XCFramework under `sdk/testpilot/build/`.

> Do not edit `.def` files directly — they are regenerated by the build script.

### Android AAR

```bash
cd sdk && ./gradlew testpilot:assembleDebug
```

Output: `sdk/testpilot/build/outputs/aar/testpilot-debug.aar`

### Installing the AAR as a local dependency

Add to your app's `build.gradle`:

```groovy
dependencies {
    def ktorVersion = "2.2.4"
    implementation files("${path_to_testpilot}/sdk/testpilot/build/outputs/aar/testpilot-debug.aar")
    implementation "com.aallam.openai:openai-client:3.1.1"
    implementation "org.jetbrains.kotlinx:kotlinx-serialization-json:1.4.1"
    implementation "io.ktor:ktor-client-core:$ktorVersion"
    implementation "io.ktor:ktor-client-cio:$ktorVersion"
    androidTestImplementation "androidx.test.uiautomator:uiautomator:2.2.0"
    androidTestImplementation "org.jetbrains.kotlin:kotlin-test"
}
```

### Installing the XCFramework as a local dependency

1. Build the XCFramework (`scripts/build_ios_sdk.sh`)
2. In Xcode, drag the repo folder (the one containing `Package.swift`) into your project
3. In your test target's **Build Phases → Link Binary With Libraries**, add `TestPilotKit`

## Key Files Reference

| File | Purpose |
|------|---------|
| `sdk/testpilot/src/commonMain/…/analyst/Analyst.kt` | Explore loop (analyze mode) |
| `sdk/testpilot/src/commonMain/…/analyst/TestAnalyst.kt` | Deterministic loop (test mode) |
| `sdk/testpilot/src/commonMain/…/analyst/HtmlReportWriter.kt` | HTML report with inline screenshots |
| `sdk/testpilot/src/commonMain/…/ai/VisionPrompt.kt` | Prompt for exploratory analysis (injects persona when present) |
| `sdk/testpilot/src/commonMain/…/ai/TestVisionPrompt.kt` | Prompt for pass/fail evaluation |
| `sdk/testpilot/src/iosMain/…/ai/CachingAIClient.kt` | FNV-1a hash cache, persists to `NSCachesDirectory` |
| `mac-app/TestPilotApp/Services/AnalysisRunner.swift` | Parses `TESTPILOT_STEP:` / `TESTPILOT_RESULT:` in real-time |
| `mac-app/TestPilotApp/Services/ArtifactManager.swift` | Downloads and verifies artifacts on first launch |
| `mac-app/TestPilotApp/Services/IOSRunner.swift` | iOS subprocess setup, bundle ID resolution, test file generation |
| `mac-app/TestPilotApp/Services/WebRunner.swift` | Web subprocess setup (invokes the JAR via bundled JRE) |
| `testpilot` | CLI entry point — `analyze`, `test`, `web-login` subcommands |

## stdout Markers

The Kotlin runners emit structured lines that both the CLI and the Mac app parse:

| Marker | Meaning |
|--------|---------|
| `TESTPILOT_STEP: <message>` | A step was executed |
| `TESTPILOT_STEP: (cached) <message>` | Step was served from cache |
| `TESTPILOT_RESULT: PASS <reason>` | Test passed |
| `TESTPILOT_RESULT: FAIL <reason>` | Test failed |
| `TESTPILOT_REPORT_PATH=<path>` | Path to the generated HTML report |
